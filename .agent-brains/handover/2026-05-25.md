# Handover Memo — 2026-05-25

`feature/hidhide-startup-ux` (layered on `feature/hidhide-integration`)

## What Was Done
Three improvements to HidHide integration:

### 1. Check-Before-Write Optimization (`3d74535e`)
`HidHideManager` now reads current HidHide state before issuing CLI write operations.
If the app path is already whitelisted / device is already blocked, no CLI call is made.
This means admin elevation is only needed on the **first** launch.

### 2. On-Demand UAC Elevation (`84178048`)
- Changed `app.manifest` from `requireAdministrator` → `asInvoker`
- `HidHideCliFallback.Run()` tries non-elevated first; on Win32 error 740 (`ERROR_ELEVATION_REQUIRED`) retries with `ShellExecute runas` (UAC popup for the subprocess only); on error 1223 (`ERROR_CANCELLED`) throws `HidHideElevationCancelledException`
- `HidHideManager.EnsureWhitelistedAsync` catches the cancellation exception at Info level (user choice — not an error)
- Added `HidHideElevationCancelledException` to `Core/Exceptions/GremlinException.cs`

### 3. Remove In-App HidHide Config Page (`96c30339`)
User decided using the native HidHide Configuration Client is simpler.
- Deleted: `HidHidePageView.axaml`, `HidHidePageViewModel.cs`, `HidHideDeviceRowViewModel.cs`, `HidHideWarningDialog.axaml`
- Removed from `MainWindowViewModel`, `App.axaml.cs` DI registration
- The 🛡 button in the main toolbar still opens the native client (IsEnabled bound to `HasHidHideClient`)
